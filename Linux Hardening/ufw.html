System Security // Firewall

UFW Configuration Guide

Uncomplicated Firewall — a friendly frontend to iptables

Debian / Ubuntu iptables / nftables

What is UFW?

Linux filters network traffic at the kernel level using iptables (or its modern successor, nftables). Writing iptables rules directly is powerful but verbose. UFW — Uncomplicated Firewall — sits on top of iptables and translates simple commands like ufw allow ssh into the correct low-level rules automatically.

UFW ships with Ubuntu and most Debian-based systems. It is installed by default but left inactive — no filtering happens until you explicitly turn it on. This guide walks you from that inactive state to a sensibly configured firewall.

⚠

If you are managing this machine over SSH, add the SSH allow rule in Step 04 before running ufw enable. When the firewall activates it enforces rules immediately — any traffic without a matching ALLOW rule is dropped on the spot, including your active session. There is no grace period.

STEP 01

Check Installation

Check whether UFW is present by querying its version. This also confirms the binary is on your PATH and executable:

terminal
sudo ufw version

A version string means UFW is installed — skip to Step 03. "command not found" means proceed to Step 02.

STEP 02

Install UFW

If UFW isn't present, install it. Running apt update first is important — it refreshes your local package index so you receive the latest available version, not a stale cached one:

terminal
sudo apt update && sudo apt install ufw -y
STEP 03

Check Current Status

Even if UFW is installed, it may not be enforcing anything yet. This command shows whether the firewall is active and lists any rules already in place. On a fresh system you will see Status: inactive:

terminal
sudo ufw status verbose

The verbose flag is more useful than a plain status — it shows your default inbound and outbound policies alongside the rule list, giving you the full picture.

STEP 04

Set Default Policies & Allow SSH

Default policies decide what happens to traffic that doesn't match any specific rule. The secure baseline is to deny all inbound and allow all outbound — nothing gets in unless you permit it, while your machine can still initiate connections freely. Add SSH before anything else so your remote session survives the next step:

terminal
# Block all unsolicited inbound connections by default sudo ufw default deny incoming # Allow your machine to reach the internet freely sudo ufw default allow outgoing # Allow SSH — "ssh" is a named service; UFW resolves it # to port 22/tcp via /etc/services automatically sudo ufw allow ssh
default deny incoming
All unsolicited inbound packets are dropped unless an explicit ALLOW rule matches them.
default allow outgoing
Your machine can freely initiate connections — package updates, DNS queries, curl, etc.
allow ssh
Expands to allow 22/tcp. Keeps your remote session alive once the firewall activates.
Why order matters
Set defaults and allow SSH before enabling — rules are enforced the moment the firewall goes live.
STEP 05

Enable the Firewall

With your defaults and SSH rule in place, activate the firewall. UFW will warn you that enabling may disrupt existing SSH connections — since you already added the rule, confirm with y:

terminal
sudo ufw enable

Verify the live ruleset. The numbered flag is important — those numbers let you delete individual rules later with ufw delete <number> without disrupting the rest:

terminal
sudo ufw status numbered
STEP 06

Opening Ports for Development

Each allow <port>/tcp creates a rule permitting inbound TCP connections on that port. Only open what your stack actually uses — every exposed port is potential attack surface. Common development server ports are listed below:

terminal
# HTTP and HTTPS — standard web server ports sudo ufw allow 80/tcp sudo ufw allow 443/tcp # Development server ports — add only what your stack uses sudo ufw allow 3000/tcp # React, Next.js, Express sudo ufw allow 8000/tcp # Django, Python http.server sudo ufw allow 8080/tcp # Java apps, Jenkins, alt-HTTP
PortProtocolTypical use
80TCPPlain HTTP — standard unencrypted web traffic
443TCPHTTPS — TLS-encrypted web traffic
3000TCPReact dev server, Express, Next.js
8000TCPDjango runserver, Python http.server
8080TCPJava applications, Jenkins, generic alt-HTTP

To remove a rule: run sudo ufw status numbered, find the rule number, then run sudo ufw delete <number>. Remaining rules renumber automatically.
